# CloudFront WAF – Web ACL

## aws_wafv2_web_acl

Creates a **WAFv2 Web ACL**.

### Resource Name

```text
armageddon_cf_waf
```

Terraform’s internal reference name.

---

## Provider Requirement

```text
provider = aws.use1
```

Uses your **us-east-1 provider alias**.

This is required because:

- CloudFront WAF resources **must be created in us-east-1**
- Even if your main infrastructure is in another region

---

## Scope

```text
scope = "CLOUDFRONT"
```

Indicates this Web ACL protects:

- CloudFront distributions

If protecting ALB/API Gateway instead:

```text
scope = "REGIONAL"
```

---

# Default Action

![[media/image2.png]]

If a request doesn’t match any blocking rule:

```text
ALLOW
```

This means:

- Only rule matches can block
- Everything else passes

---

# Visibility Configuration (ACL Level)

![[media/image3.png]]

## CloudWatch Metrics

```text
cloudwatch_metrics_enabled = true
```

Publishes WAF metrics to **CloudWatch**.

---

## Metric Name

Defines the CloudWatch metric namespace/name for this ACL.

---

## Sampled Requests

```text
sampled_requests_enabled = true
```

WAF stores sampled requests for:

- Troubleshooting
- Reviewing attack attempts
- Console inspection

---

# Managed Rule Group

![[media/image4.png]]

Adds one rule to the ACL:

```text
AWSManagedRulesCommonRuleSet
```

This AWS-managed rule group covers:

- Common web exploits
- Suspicious request patterns
- Generic Layer 7 attacks
- Basic bot filtering patterns

---

## Rule Priority

```text
priority = 1
```

Lower number = evaluated first.

With one rule, it effectively runs immediately.

---

## override_action

```text
override_action {
  none {}
}
```

Meaning:

- Do **not override**
- Let AWS managed rule group use its default behavior

Typically:

- Block malicious traffic
- Allow legitimate traffic

---

### Alternative: Count Mode

If using:

```text
override_action {
  count {}
}
```

Then:

- Rule matches are logged
- Requests are **not blocked**
- Common during testing phases

---

# Statement Block

![[media/image5.png]]

Defines the match logic:

```text
managed_rule_group_statement {
  name        = "AWSManagedRulesCommonRuleSet"
  vendor_name = "AWS"
}
```

This tells WAF to use AWS’s managed rule group.

---

# Visibility Configuration (Rule Level)

![[media/image6.png]]

Same idea as ACL-level visibility, but specific to this rule:

- Separate CloudWatch metrics
- Separate sampled request tracking

This allows:

- Fine-grained monitoring
- Understanding which rule group is triggering

---

# Evaluation Flow

```text
Incoming Request
      ↓
WAF Web ACL (CLOUDFRONT scope)
      ↓
Rule Priority 1:
  AWSManagedRulesCommonRuleSet
      ↓
Match?
   ↓ Yes → Block (default rule action)
   ↓ No  → Default Action = Allow
```

---

# Architecture Context

```text
User
   ↓
CloudFront
   ↓
CloudFront WAF (Global Scope)
   ↓
Origin (ALB)
```

This protects traffic:

- Before it reaches ALB
- At the CloudFront edge layer

---

# Design Patterns Used

|Pattern|Purpose|
|---|---|
|us-east-1 provider alias|Required for CloudFront WAF|
|CLOUDFRONT scope|Global edge protection|
|Managed rule group|Prebuilt AWS protections|
|override_action none|Use AWS defaults|
|Count mode option|Safe testing phase|
|CloudWatch metrics|Observability|
|Sampled requests|Forensics & debugging|

---

# Mental Model

- CloudFront WAF must live in us-east-1
- Default = allow unless blocked
- Managed rules provide baseline security
- Metrics + sampling enable visibility
- Count mode = monitor only
- None override = enforce protection
